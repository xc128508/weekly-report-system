# 实习生周报导入接口说明

本文档用于对接“文档/docx 系统 -> 实习生周报系统”的周报导入流程。

对方系统负责上传 docx、解析/转换周报内容、提交数据；本系统负责接口鉴权、匹配实习生账号、保存周报，并在实习生端和老板端展示。

## 一、系统职责

对方文档系统:

- 负责 docx 上传、解析、转换。
- 负责在对方系统内填写/编辑周报内容。
- 负责调用本系统导入接口提交结构化周报。

本实习生周报系统:

- 已恢复实习生端本地填写、保存草稿、提交和修改周报功能。
- 仍保留外部文档系统导入周报能力，导入后的周报也可在实习生端和老板端查看。
- 自动周报草稿可基于本系统每日任务生成；外部 docx 解析/转换流程仍由对方系统负责。
- 只负责鉴权、存储、查看、老板端反馈。
- 通过 `POST /api/weekly-reports/import` 接收对方系统提交的周报。

## 二、接口地址

健康检查:

```http
GET /api/weekly-reports/import/health
```

周报导入:

```http
POST /api/weekly-reports/import
```

请求格式:

```http
Content-Type: application/json
```

## 三、api-token.txt 配置

本系统使用服务器本地文件 `data/api-token.txt` 保存导入接口 token。

生产服务器示例目录:

```bash
/var/www/weekly-report-system/data/api-token.txt
```

文件内容:

```txt
your-api-token-here
```

注意:

- `api-token.txt` 文件里只放 token 本身，不要加空格、注释或 JSON。
- token 由本系统保存，对方系统调用接口时通过请求头传入。

推荐权限:

```bash
chmod 600 data/api-token.txt
chown root:root data/api-token.txt
```

检查命令:

```bash
ls -lh data/api-token.txt
cat data/api-token.txt
```

期望类似:

```txt
-rw------- 1 root root 65 Jul  6 10:03 data/api-token.txt
your-api-token-here
```

## 四、鉴权方式

推荐使用请求头:

```http
x-api-token: your-api-token-here
```

也兼容:

```http
Authorization: Bearer your-api-token-here
```

未配置 token 时，本系统返回:

```json
{
  "success": false,
  "message": "周报导入 API token 未配置，请先在系统 data/api-token.txt 中配置 token"
}
```

token 错误时，本系统返回:

```json
{
  "success": false,
  "message": "API token 无效"
}
```

## 五、必填规则

必须满足以下三类字段:

用户标识至少传一个:

- `username`（推荐，等于本系统账号）
- `realName`
- `userId`（可选兼容字段）

周报周期必须传:

- `weekStart` / `week_start` / `startDate`
- `weekEnd` / `week_end` / `endDate`

周报内容至少传一个:

- `workContent`
- `achievements`
- `nextPlan`
- `content`
- `markdown`
- `summary`

建议优先传 `username`，也就是本系统账号管理里的“账号”。不需要传 `userId`；`userId` 仅作为可选兼容字段。

## 七、字段兼容表

用户字段:

## 八、存储规则

本系统按以下条件判断新增或更新:

```txt
userId + weekStart + weekEnd
```

如果同一实习生同一周期已经有周报:

- 更新原周报。
- 返回 `mode: "update"`。

如果没有:

- 新增周报。
- 返回 `mode: "create"`。

导入后的周报会出现在:

- 实习生端: `我的周报`
- 老板端: `周报管理`

## 十、错误响应

token 错误:

```json
{
  "success": false,
  "message": "API token 无效"
}
```

缺少用户标识:

```json
{
  "success": false,
  "message": "缺少用户标识：请传 userId、username 或 realName"
}
```

找不到实习生:

```json
{
  "success": false,
  "message": "未找到实习生用户：intern001"
}
```

姓名重复:

```json
{
  "success": false,
  "message": "用户姓名匹配到多个账号，请改传 userId 或 username：实习生甲"
}
```

缺少周期:

```json
{
  "success": false,
  "message": "缺少周报周期：请传 weekStart/week_start 和 weekEnd/week_end，格式 YYYY-MM-DD"
}
```

缺少内容:

```json
{
  "success": false,
  "message": "缺少周报内容：请至少传 workContent、achievements、nextPlan、content、markdown 或 summary 中的一个字段"
}
```

## 十三、对方系统联调 Checklist

- 已拿到本系统提供的接口 token。
- 请求头包含 `x-api-token` 或 `Authorization: Bearer your-api-token-here`。
- `Content-Type` 是 `application/json`。
- 优先传 `username`，即本系统账号管理里的“账号”。
- 日期格式是 `YYYY-MM-DD`。
- 同一实习生同一周期重复提交时，确认是更新而不是新增。
- 导入成功后，在本系统实习生端“我的周报”可查看。
- 导入成功后，在本系统老板端“周报管理”可查看。

## 十四、安全说明

- 替换 token 后，对方系统需要同步更新请求头中的 token。
